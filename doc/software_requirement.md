# ソフトウェア要件定義書 (Software Requirements Specification)

## 1. プロジェクト概要

### 1.1 プロジェクト名
Knowledge Graph LLMs

### 1.2 目的
テキストデータから自動的にナレッジグラフを生成し、視覚的に表示するWebアプリケーションを提供する。

### 1.3 スコープ
- テキストからのエンティティ抽出
- エンティティ間の関係性抽出
- インタラクティブなグラフ可視化
- グラフ履歴管理
- 複数LLMモデル対応

### 1.4 対象ユーザー
- 研究者
- データアナリスト
- ナレッジマネージャー
- ドキュメント分析担当者

## 2. 機能要件

### 2.1 テキスト入力機能 (FR-001)

#### FR-001-1: ファイルアップロード
- **優先度**: 必須
- **説明**: ユーザーがテキストファイルをアップロードできる
- **実装箇所**: `app.py:160-177`
- **受入基準**:
  - `.txt` 形式のファイルを受け入れる
  - UTF-8エンコーディングで読み込む
  - ファイルサイズ制限はStreamlitデフォルト（200MB）
- **制約条件**:
  - 1ファイルずつ処理
  - テキストファイルのみ対応

#### FR-001-2: テキスト直接入力
- **優先度**: 必須
- **説明**: ユーザーがテキストエリアに直接入力できる
- **実装箇所**: `app.py:178-194`
- **受入基準**:
  - 複数行テキスト入力可能
  - 入力エリア高さ200px
  - リアルタイム入力可能
- **制約条件**:
  - テキスト長の上限はStreamlitデフォルト

#### FR-001-3: 入力方法切り替え
- **優先度**: 必須
- **説明**: ファイルアップロードとテキスト入力を切り替えられる
- **実装箇所**: `app.py:131-143`
- **受入基準**:
  - ラジオボタンで選択
  - 選択に応じてUIが切り替わる
  - 前回の選択状態を記憶
- **技術仕様**:
  - Session Stateで状態管理

### 2.2 LLMモデル選択機能 (FR-002)

#### FR-002-1: モデル選択
- **優先度**: 必須
- **説明**: 複数のOpenAI LLMモデルから選択できる
- **実装箇所**: `app.py:122-129`
- **対応モデル**:
  - gpt-4.1-mini (デフォルト)
  - gpt-4.1-nano
  - gpt-4.1
  - gpt-4o-mini
  - gpt-4o
- **受入基準**:
  - ドロップダウンで選択
  - デフォルトは gpt-4.1-mini
  - 選択したモデルでグラフ生成

### 2.3 ナレッジグラフ生成機能 (FR-003)

#### FR-003-1: エンティティ抽出
- **優先度**: 必須
- **説明**: テキストからエンティティ（ノード）を抽出
- **実装箇所**: `generate_knowledge_graph.py:23-28`
- **技術仕様**:
  - LangChain LLMGraphTransformer使用
  - 非同期処理 (asyncio)
- **受入基準**:
  - テキストから意味のあるエンティティを抽出
  - 各ノードにid と type を付与
  - 重複エンティティの排除

#### FR-003-2: 関係性抽出
- **優先度**: 必須
- **説明**: エンティティ間の関係性（エッジ）を抽出
- **実装箇所**: `generate_knowledge_graph.py:23-28`
- **受入基準**:
  - エンティティ間の意味的な関係を抽出
  - 関係性にtype（関係の種類）を付与
  - source と target を明確に定義

#### FR-003-3: グラフ可視化
- **優先度**: 必須
- **説明**: 抽出したグラフをインタラクティブに表示
- **実装箇所**: `generate_knowledge_graph.py:41-115`
- **技術仕様**:
  - PyVis Network使用
  - 有向グラフ
  - 背景色: #222222（ダーク）
  - フォント色: white
  - フィルターメニュー有効
- **受入基準**:
  - ノードをドラッグ可能
  - エッジにラベル表示
  - ズーム・パン操作可能
  - ノード色分け（typeごと）

#### FR-003-4: 物理演算レイアウト
- **優先度**: 必須
- **説明**: グラフを見やすく自動配置
- **実装箇所**: `generate_knowledge_graph.py:92-105`
- **技術仕様**:
  - ForceAtlas2Based ソルバー
  - パラメータ:
    - gravitationalConstant: -100
    - centralGravity: 0.01
    - springLength: 200
    - springConstant: 0.08
    - minVelocity: 0.75
- **受入基準**:
  - ノードが重ならない
  - 関連ノードが近くに配置
  - 視覚的に理解しやすい

### 2.4 テキスト要約機能 (FR-004)

#### FR-004-1: 自動要約
- **優先度**: 必須
- **説明**: 入力テキストを自動で要約
- **実装箇所**: `generate_knowledge_graph.py:31-38`
- **受入基準**:
  - 最大25文字
  - ファイル名に使用可能な形式
  - 改行を除去
- **用途**:
  - グラフファイル名生成
  - 履歴一覧での識別

### 2.5 ファイル保存機能 (FR-005)

#### FR-005-1: HTML保存
- **優先度**: 必須
- **説明**: 生成したグラフをHTMLファイルで保存
- **実装箇所**: `generate_knowledge_graph.py:107-114`
- **ファイル命名規則**:
  - 形式: `{timestamp}_{summary}_{model}.html`
  - timestamp: `YYYYMMDD_HHMMSS`
  - summary: テキスト要約（25文字以内）
  - model: 使用したLLMモデル名
- **保存先**: `out/` ディレクトリ
- **受入基準**:
  - ブラウザで単独表示可能
  - インタラクティブ機能保持

#### FR-005-2: JSON保存
- **優先度**: 必須
- **説明**: ノードとリレーションシップをJSON形式で保存
- **実装箇所**: `generate_knowledge_graph.py:141-156`
- **データ構造**:
```json
{
  "nodes": [
    {"id": "string", "type": "string"}
  ],
  "relationships": [
    {"source": "string", "target": "string", "type": "string"}
  ]
}
```
- **保存先**: `out/` ディレクトリ（HTMLと同名で拡張子 `.json`）
- **受入基準**:
  - UTF-8エンコーディング
  - インデント付き（読みやすさ）
  - ensure_ascii=False（日本語保持）

### 2.6 グラフ履歴管理機能 (FR-006)

#### FR-006-1: 履歴一覧表示
- **優先度**: 必須
- **説明**: 過去に生成したグラフを一覧表示
- **実装箇所**: `app.py:145-156`
- **受入基準**:
  - ドロップダウンで選択
  - 表示形式: `{summary}_{model}`
  - 新しい順にソート
- **データソース**: `out/` ディレクトリの `.html` ファイル

#### FR-006-2: 履歴からグラフ表示
- **優先度**: 必須
- **説明**: 選択したグラフを再表示
- **実装箇所**: `app.py:196-206`
- **受入基準**:
  - ノード・リレーション一覧表示
  - グラフ可視化表示
  - JSONデータ読み込み
  - 元の状態を完全再現

### 2.7 ノード・リレーション表示機能 (FR-007)

#### FR-007-1: 一覧表示
- **優先度**: 必須
- **説明**: ノードとリレーションをリスト表示
- **実装箇所**: `app.py:65-111`
- **表示形式**:
  - ノード: `{id} ({type})`
  - リレーション: `{source} -[{type}]-> {target}`
- **受入基準**:
  - スクロール可能（高さ300px）
  - 視認性の高いレイアウト

#### FR-007-2: クリップボードコピー
- **優先度**: 必須
- **説明**: ノード・リレーション情報をコピー
- **実装箇所**: `app.py:92-108`
- **受入基準**:
  - Copyボタンクリックでコピー
  - 複数ブラウザ対応
  - テキスト全体をコピー
- **技術仕様**:
  - UUID使用（ID競合回避）
  - 3段階フォールバック:
    1. Clipboard API
    2. window.clipboardData (IE)
    3. document.execCommand (レガシー)

### 2.8 エラーハンドリング機能 (FR-008)

#### FR-008-1: 部分的失敗の許容
- **優先度**: 必須
- **説明**: ノード・エッジ追加時のエラーを許容
- **実装箇所**: `generate_knowledge_graph.py:79-89`
- **動作**:
  - エラー発生時は該当要素をスキップ
  - 処理を継続
  - try-except-continue パターン
- **受入基準**:
  - 一部のノードが無効でもグラフ生成完了
  - 有効なノードのみ表示

#### FR-008-2: ファイル保存エラー処理
- **優先度**: 必須
- **説明**: ファイル保存失敗時の適切な処理
- **実装箇所**: `generate_knowledge_graph.py:108-114`, `143-156`
- **動作**:
  - エラーメッセージをコンソール出力
  - 処理は継続（可能な限り）
- **受入基準**:
  - エラー内容を明示
  - ユーザーに適切なフィードバック

## 3. 非機能要件

### 3.1 パフォーマンス (NFR-001)

#### NFR-001-1: 処理時間
- **要件**: グラフ生成時間は合理的な範囲内
- **目標**: 約9秒（OpenAI API依存）
- **制約**: API応答時間に依存
- **測定方法**: 生成ボタンクリックから表示完了まで

#### NFR-001-2: 推奨データ量
- **ノード数**: 50-200
- **エッジ数**: 100-300
- **理由**: 可視化の視認性とパフォーマンスのバランス

### 3.2 スケーラビリティ (NFR-002)

#### NFR-002-1: ユーザー数
- **想定**: 単一ユーザー
- **実装**: Streamlit単体（セッション管理なし）
- **拡張性**: 複数ユーザー対応は将来検討

#### NFR-002-2: データ保存
- **方式**: ローカルファイルシステム
- **制約**: ディスク容量に依存
- **履歴数**: 制限なし（手動削除）

### 3.3 セキュリティ (NFR-003)

#### NFR-003-1: API キー管理
- **要件**: OpenAI API キーの安全な管理
- **実装**: 環境変数 (`.env` ファイル)
- **実装箇所**: `generate_knowledge_graph.py:16-19`
- **受入基準**:
  - `.env` をGit管理外
  - ハードコーディング禁止

#### NFR-003-2: ファイルアクセス制御
- **要件**: 安全なファイル操作
- **実装**: `os.path.join` 使用
- **受入基準**:
  - パストラバーサル対策
  - 出力ディレクトリ限定

### 3.4 ユーザビリティ (NFR-004)

#### NFR-004-1: UI言語
- **要件**: 日本語UI
- **実装状況**: 完全対応
- **範囲**: ラベル、メッセージ、コメント

#### NFR-004-2: フィードバック
- **要件**: 処理状態の明示
- **実装**:
  - スピナー表示（処理中）
  - 成功メッセージ（完了時）
- **実装箇所**: `app.py:169`, `173`, `186`, `190`

#### NFR-004-3: レスポンシブデザイン
- **要件**: 画面サイズに応じた表示
- **実装**: Wide layout
- **制約**: デスクトップ環境推奨

### 3.5 保守性 (NFR-005)

#### NFR-005-1: コード品質
- **要件**: 読みやすく保守しやすいコード
- **実装**:
  - 関数分割
  - 日本語コメント
  - docstring完備
- **主要コメント箇所**:
  - `app.py:11-62` (関数説明)
  - `generate_knowledge_graph.py:23-129` (処理説明)

#### NFR-005-2: 依存関係管理
- **要件**: 明確な依存関係定義

#### NFR-006-1: ブラウザ対応
- **要件**: モダンブラウザで動作

## 4. システム制約

### 4.1 技術スタック
- **言語**: Python 3.8+
- **フレームワーク**: Streamlit
- **LLMライブラリ**: LangChain
- **可視化**: PyVis
- **LLMプロバイダ**: OpenAI

### 4.2 外部依存
- **OpenAI API**: 必須
- **インターネット接続**: API呼び出しに必要
- **CDN**: PyVis リソース（`cdn_resources='remote'`）

### 4.3 ファイルシステム
- **出力ディレクトリ**: `out/`
- **環境設定**: `.env`
- **権限**: 読み書き権限必要

## 5. データ要件

### 5.1 入力データ
- **形式**: プレーンテキスト
- **エンコーディング**: UTF-8
- **最大サイズ**: 200MB（Streamlitデフォルト）
- **言語**: 制限なし（LLM対応範囲内）

### 5.2 出力データ
- **HTML**: PyVis生成、ブラウザ表示可能
- **JSON**: UTF-8、インデント付き、日本語対応
- **命名規則**: タイムスタンプベース（一意性保証）

### 5.3 メタデータ
- **タイムスタンプ**: `YYYYMMDD_HHMMSS`
- **要約**: 最大25文字
- **モデル名**: 選択したLLM

## 6. 運用要件

### 6.1 起動方法
```bash
streamlit run app.py
```

### 6.2 環境設定
- `.env` ファイルに `OPENAI_API_KEY` 設定
- `out/` ディレクトリは自動作成

### 6.3 ログ・監視
- **標準出力**: グラフ保存完了メッセージ
- **エラー出力**: ファイル保存エラー、JSON保存エラー
